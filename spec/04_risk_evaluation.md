# RIO Protocol Specification: 04 Risk Evaluation

## 1. Protocol Name

RIO Protocol Step 04: Risk Evaluation

## 2. Purpose

This protocol step is responsible for assessing the potential risk associated with a `canonical_request`. Its primary purpose is to analyze the request against a set of predefined risk factors, calculate a quantitative risk score, and produce a qualitative risk level. Based on this assessment, it generates a recommendation for how the request should be handled in subsequent steps of the RIO decision chain. This ensures that actions with a higher potential for negative impact are subjected to greater scrutiny and control, such as requiring explicit human authorization.

## 3. Scope

This protocol's scope is strictly limited to the evaluation of a single `canonical_request` and the generation of a corresponding `risk_evaluation` record. It is responsible for interpreting the data within the `risk_context` field of the request, applying scoring logic, and determining an appropriate recommendation.

What is explicitly out of scope:
-   The enforcement of policy constraints.
-   The gathering of the initial risk context data (which is part of the Intake step).
-   The process of obtaining human authorization.
-   The execution of the requested action.

## 4. Inputs

The primary input for this protocol step is the `canonical_request` record generated by the preceding "Canonical Request Formation" step.

| Field             | Type        | Required | Description                                                                                             |
| ----------------- | ----------- | -------- | ------------------------------------------------------------------------------------------------------- |
| `request_id`      | `UUID`      | Yes      | The unique identifier for the request.                                                                  |
| `requested_by`    | `String`    | Yes      | Identifier for the AI agent or system initiating the action.                                            |
| `action_type`     | `String`    | Yes      | The specific type of action being requested (e.g., `wire_transfer`, `api_call`).                        |
| `target`          | `String`    | Yes      | The resource or entity the action is directed at.                                                       |
| `parameters`      | `JSON`      | Yes      | A structured object containing the specific parameters for the action.                                  |
| `business_reason` | `String`    | Yes      | A human-readable justification for the action.                                                          |
| `risk_context`    | `JSON`      | Yes      | Data used for risk assessment, such as transaction amount, user location, or historical frequency.     |
| `policy_context`  | `JSON`      | Yes      | Information relevant to policy evaluation, such as user roles or applicable regulatory constraints. |

## 5. Outputs

The sole output of this protocol step is a new `risk_evaluation` record, which conforms to the `risk_evaluation.json` schema.

| Field                 | Type     | Description                                                                                                |
| --------------------- | -------- | ---------------------------------------------------------------------------------------------------------- |
| `risk_evaluation_id`  | `UUID`   | The unique identifier for this specific risk evaluation.                                                   |
| `request_id`          | `UUID`   | The identifier of the `canonical_request` that was evaluated.                                              |
| `risk_level`          | `Enum`   | A qualitative assessment of the risk (`low`, `medium`, `high`, `critical`).                                |
| `risk_score`          | `Integer`| A quantitative score from 0-100 representing the calculated risk.                                            |
| `risk_factors`        | `JSON`   | An object detailing the specific factors that contributed to the score (e.g., `amount_factor`, `velocity_factor`). |
| `policy_flags`        | `Array`  | A list of policy-related flags triggered by the request (e.g., `requires_cfo_approval`).                 |
| `recommendation`      | `Enum`   | The recommended next step (`approve`, `require_authorization`, `deny`, `escalate`).                        |

## 6. Required Fields

For the Risk Evaluation step to proceed, the following fields from the `canonical_request` input MUST be present and valid. Failure to meet this requirement SHALL result in a processing failure.

| Field            | Reason                                                              |
| ---------------- | ------------------------------------------------------------------- |
| `request_id`     | Essential for linking the evaluation back to the original request.  |
| `action_type`    | Determines the set of risk rules to apply.                          |
| `parameters`     | Contains the core data points (e.g., amount) needed for scoring.    |
| `risk_context`   | Provides the contextual data required by the risk scoring engine.   |
| `policy_context` | Needed to identify which policies might be flagged.                 |

## 7. Processing Steps

1.  The protocol processor SHALL receive a `canonical_request` record.
2.  The processor MUST validate the presence and basic format of all fields listed in Section 6. If validation fails, the processor SHALL enter a failure condition as defined in Section 9.
3.  A new `risk_evaluation` record SHALL be created, and a unique `risk_evaluation_id` SHALL be generated.
4.  The `request_id` from the input record MUST be copied to the `risk_evaluation` record.
5.  The risk scoring engine SHALL analyze the `risk_context`, `action_type`, and `parameters` to identify and score individual risk factors. Each factor (e.g., transaction amount, beneficiary reputation, time of day) is assigned a score.
6.  The individual factor scores SHALL be aggregated into a final, normalized `risk_score` between 0 and 100.
7.  The `risk_score` SHALL be used to determine the `risk_level` based on the thresholds defined in the Decision Logic (Section 8).
8.  The `policy_context` SHALL be analyzed to identify any applicable `policy_flags` that must be attached to the evaluation.
9.  A final `recommendation` SHALL be determined based on the combination of `risk_level` and `policy_flags` as per the Decision Logic.
10. The completed `risk_evaluation` record, containing all the fields described in Section 5, MUST be emitted for consumption by the next protocol step.

## 8. Decision Logic

The decision logic is composed of two parts: Risk Level Assignment and Recommendation Generation.

**Risk Level Assignment**

The `risk_level` is determined by the calculated `risk_score` according to the following thresholds:

| Risk Score Range | Risk Level  |
| ---------------- | ----------- |
| 0 - 30           | `low`       |
| 31 - 70          | `medium`    |
| 71 - 90          | `high`      |
| 91 - 100         | `critical`  |

**Recommendation Generation**

The `recommendation` is determined by a ruleset that considers both the `risk_level` and any `policy_flags`.

| Condition                                                              | Recommendation             |
| ---------------------------------------------------------------------- | -------------------------- |
| IF `risk_level` is `critical`                                          | `deny`                     |
| IF `risk_level` is `high`                                              | `require_authorization`    |
| IF `policy_flags` contains `requires_human_approval`                   | `require_authorization`    |
| IF `risk_level` is `medium` AND no prohibitive flags are present       | `approve`                  |
| IF `risk_level` is `low`                                               | `approve`                  |
| IF evaluation logic cannot resolve a recommendation                    | `escalate`                 |

## 9. Failure Conditions

This step can fail under specific conditions. Each failure MUST be logged with the appropriate error code.

| Error Code | Trigger                                                                                                | Required Action                                                                                              |
| ---------- | ------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------ |
| `E0401`    | A required field from Section 6 is missing, null, or malformed in the `canonical_request`.             | The request MUST be rejected. A failure record should be logged, and the process terminated for this request. |
| `E0402`    | The risk scoring engine is unable to process the `risk_context` data (e.g., invalid format, unknown values). | The request SHOULD be moved to an `escalate` state for manual review by a risk operations team.              |
| `E0403`    | The risk scoring rules for the given `action_type` are not defined or fail to execute.                 | The request MUST be moved to an `escalate` state. An alert should be sent to the system administrators.     |

## 10. Security Considerations

-   **Integrity of Inputs**: The `canonical_request` record MUST be cryptographically signed or otherwise protected to ensure it has not been tampered with since its creation. The risk evaluation process must validate this integrity before processing.
-   **Confidentiality of Logic**: The specific thresholds and weighting of risk factors within the scoring engine are sensitive and SHOULD be protected from unauthorized disclosure to prevent adversaries from reverse-engineering the logic to bypass controls.
-   **Auditability**: The resulting `risk_evaluation` record MUST be stored in an immutable log to provide a verifiable record of the risk assessment outcome.
-   **Denial of Service**: The processing logic SHOULD be designed to handle malformed inputs gracefully to prevent denial-of-service attacks that could exhaust system resources.

## 11. Audit Requirements

To ensure transparency and accountability, the following audit requirements MUST be met:
-   A complete copy of every `risk_evaluation` record generated MUST be stored in a secure, immutable audit ledger.
-   The stored record MUST be linked to its corresponding `canonical_request` record.
-   The version of the risk scoring model and ruleset used for the evaluation SHOULD be recorded alongside the evaluation itself to allow for accurate reconstruction of the decision process at a later date.
-   Access to the risk evaluation logic and historical evaluation records MUST be strictly controlled and logged.

## 12. Dependencies

-   **Upstream**: This protocol step is dependent on the successful completion of **Step 03: Canonical Request Formation**. It cannot proceed without a valid `canonical_request` record.
-   **Downstream**: The output of this step, the `risk_evaluation` record, is a primary input for **Step 05: Policy Constraints**. The recommendation provided directly influences how the policy engine processes the request.

## 13. Example Flow

This example follows the scenario of an AI agent requesting a $48,250 wire transfer.

1.  **Input Received**: The Risk Evaluation processor receives a `canonical_request` with the following key values:
    -   `request_id`: `8a1b2c3d-4e5f-6a7b-8c9d-0e1f2a3b4c5d`
    -   `action_type`: `wire_transfer`
    -   `parameters`: `{"amount": 48250.00, "currency": "USD", "beneficiary": "Meridian Industrial Supply", "invoice_id": "INV-2026-03-1147"}`
    -   `risk_context`: `{"transaction_amount_usd": 48250.00, "beneficiary_trust_level": "new", "agent_historical_behavior": "normal"}`
    -   `policy_context`: `{"payment_authority_limit": 10000.00, "approver_role": "cfo"}`

2.  **Processing**: The risk engine begins its evaluation.
    -   A new `risk_evaluation` record is created with `risk_evaluation_id`: `f1e2d3c4-b5a6-7b8c-9d0e-1f2a3b4c5d6e`.
    -   The engine scores the risk factors:
        -   `amount_factor`: The amount of $48,250 is significantly above the typical threshold. Score: 85/100.
        -   `beneficiary_factor`: "Meridian Industrial Supply" is a new, untrusted beneficiary. Score: 70/100.
        -   `contextual_factor`: Other factors are normal. Score: 20/100.
    -   The weighted average of these factors results in a final `risk_score`.
        -   `risk_score`: `82`

3.  **Decision Logic Applied**:
    -   With a `risk_score` of 82, the `risk_level` is set to `high` (based on the 71-90 threshold).
    -   The `policy_context` indicates that any payment over $10,000 requires special approval, so a `policy_flags` array is populated: `["requires_cfo_approval"]`.
    -   Based on the `high` risk level, the decision logic determines the `recommendation`.
        -   `recommendation`: `require_authorization`

4.  **Output Emitted**: The final `risk_evaluation` record is produced and passed to the next step.
    -   `risk_evaluation_id`: `f1e2d3c4-b5a6-7b8c-9d0e-1f2a3b4c5d6e`
    -   `request_id`: `8a1b2c3d-4e5f-6a7b-8c9d-0e1f2a3b4c5d`
    -   `risk_level`: `high`
    -   `risk_score`: `82`
    -   `risk_factors`: `{"amount_factor": 85, "beneficiary_factor": 70, "contextual_factor": 20}`
    -   `policy_flags`: `["requires_cfo_approval"]`
    -   `recommendation`: `require_authorization`
