# Audit Ledger Protocol

**Version:** 1.0.0
**Status:** Core Specification
**Category:** Data Structure / Protocol Stage

---

## Overview

The Audit Ledger Protocol defines the append-only ledger model used by Stage 8 (Audit Ledger) of the Governed Execution Protocol. The ledger records every receipt generated by the protocol as an immutable, hash-linked entry. It provides the foundation for audit, forensic reconstruction, compliance verification, and governance learning.

---

## Ledger Model

The audit ledger is an ordered, append-only sequence of ledger entries. Each entry records a single receipt and links to the previous entry by hash, forming a tamper-evident chain.

### Ledger Entry Structure

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `ledger_entry_id` | string (UUID v4) | Yes | Unique identifier for this ledger entry |
| `receipt_id` | string (UUID v4) | Yes | Reference to the receipt recorded by this entry |
| `receipt_hash` | string | Yes | SHA-256 hash of the canonical JSON representation of the referenced receipt |
| `previous_ledger_hash` | string | Yes | SHA-256 hash of the immediately preceding ledger entry. For the genesis entry, this value is a defined constant (`0x0000...0000`, 64 hex characters) |
| `timestamp` | integer | Yes | UTC Unix timestamp in milliseconds when this entry was appended |
| `ledger_signature` | string | Yes | ECDSA-secp256k1 signature over the canonical JSON representation of this entry (excluding the `ledger_signature` field itself) |

## Fundamental Properties

The audit ledger enforces the following properties at the protocol level. These are not configurable and cannot be relaxed by policy.

### Append-Only

The ledger accepts only append operations. No update, delete, reorder, or overwrite operation is permitted on any existing entry. Once an entry is appended, it is permanent. The Ledger Service exposes a single write operation (`append`) and no modification operations.

### Hash-Linked Chain

Each ledger entry contains a `previous_ledger_hash` field that holds the SHA-256 hash of the immediately preceding entry. This creates a hash chain from the genesis entry forward. Any modification to a historical entry changes its hash, which breaks the `previous_ledger_hash` reference in the next entry and all subsequent entries.

### Receipts as Entries

Ledger entries are receipts. Every receipt generated by Stage 7 (Receipt / Attestation) must be recorded as a ledger entry. The ledger does not store raw requests, intermediate processing artifacts, or any data other than receipts and their hash-chain metadata.

### No Silent Gaps

Every receipt must produce a corresponding ledger entry. A receipt without a ledger entry, or a ledger entry without a valid receipt, constitutes a protocol integrity violation. The hash chain makes gaps detectable — a missing entry breaks the chain.

### No Update or Delete

The ledger provides no mechanism to modify or remove entries after they are appended. This property is enforced at the API level (the Ledger Service does not implement update or delete endpoints), at the storage level (the underlying storage is configured for append-only writes), and at the protocol level (INV-04 requires an unbroken hash chain).

## Audit and Forensic Reconstruction

The ledger supports the following audit operations:

**Full chain verification.** An auditor recomputes all hashes from the genesis entry forward and verifies that each `previous_ledger_hash` matches the computed hash of the preceding entry. Any break in the chain indicates tampering or data loss.

**Receipt verification.** For each ledger entry, the auditor retrieves the referenced receipt and verifies that the `receipt_hash` in the ledger entry matches the SHA-256 hash of the receipt's canonical JSON representation.

**Signature verification.** The auditor verifies the `ledger_signature` on each entry using the Ledger Service's public key (obtained from the key registry).

**Temporal ordering.** The auditor verifies that all entry timestamps are monotonically increasing. A timestamp that precedes the previous entry's timestamp indicates a sequencing violation.

**Decision chain reconstruction.** Starting from any ledger entry, an auditor can reconstruct the full decision chain by following the `receipt_id` to the receipt, then following the receipt's `canonical_intent_id` to the intent, and the `authorization_id` to the authorization record. This provides a complete, verifiable history of what was requested, who authorized it, what was executed, and what was recorded.

---

## Genesis Entry

The first entry in the ledger is the genesis entry. It uses a defined constant (`0x0000...0000`, 64 hex characters) as its `previous_ledger_hash`. This is the only entry permitted to use this value. The genesis entry establishes the root of the hash chain.

---

## Failure Handling

If the Ledger Service cannot append an entry (storage failure, signature failure, hash computation failure), the protocol pipeline halts. No further executions proceed until the ledger is restored to a consistent state. A failed ledger append is treated as a system integrity failure, not a recoverable error.

This behavior is consistent with the fail-closed guarantee: the system defaults to denying execution when it cannot verify that the audit trail is intact.
