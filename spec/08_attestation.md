# RIO Protocol Specification: 08_attestation

## 1. Protocol Name

RIO Protocol Step 08: Attestation

## 2. Purpose

This protocol step provides cryptographic proof that the entire decision chain for a given AI-initiated action is authentic, complete, and has not been altered. It accomplishes this by creating a final, signed attestation record that cryptographically links all previous records in the sequence. The existence of a valid `attestation_record` serves as a non-repudiable guarantee of the process integrity up to this point.

## 3. Scope

This protocol is responsible for the generation of the `attestation_record.json` schema. This includes:
- Retrieving all prior records in the decision chain (Canonical Request, Risk Evaluation, Authorization, Execution).
- Computing cryptographic hashes of each individual record.
- Creating a sequential `chain_hash` that binds the records in order.
- Performing a series of verification checks to ensure consistency and validity.
- Producing one or more cryptographic signatures over the entire record set.

**Out of Scope:**
- The long-term storage or archival of the `attestation_record`. This is handled by the subsequent Audit Ledger step.
- The specific implementation details of the underlying cryptographic libraries, although algorithm standards are specified.
- The transmission or distribution of the generated attestation record.

## 4. Inputs

This protocol step receives the `execution_record` as its primary trigger and references all preceding records in the chain for the given `request_id`.

| Field | Type | Required | Description |
|---|---|---|---|
| `execution_record` | JSON Object | Yes | The full record from the preceding Execution step. |
| `canonical_request` | JSON Object | Yes | Referenced via `request_id`. The initial request. |
| `risk_evaluation` | JSON Object | Yes | Referenced via `request_id`. The risk assessment. |
| `authorization_record` | JSON Object | Yes | Referenced via `request_id`. The human authorization decision. |

## 5. Outputs

This protocol step produces a single, self-contained `attestation_record`.

| Field | Type | Description |
|---|---|---|
| `attestation_record` | JSON Object | A structured JSON object conforming to `attestation_record.json` containing the hashes, verification checks, and signatures for the entire decision chain. |

## 6. Required Fields

For the attestation process to begin, the following fields, derived from the input records, MUST be present and valid.

| Field | Source Record | Description |
|---|---|---|
| `request_id` | `execution_record` | The unique identifier linking the entire chain. |
| `authorization_id` | `execution_record` | The identifier for the authorization record. |
| `execution_id` | `execution_record` | The identifier for the execution record itself. |

## 7. Processing Steps

1.  The protocol SHALL be initiated upon the successful completion of the Execution step (Step 07) and receipt of a valid `execution_record`.
2.  The system SHALL retrieve the complete `canonical_request`, `risk_evaluation`, `authorization_record`, and `execution_record` documents using their respective IDs referenced in the chain.
3.  The system SHALL compute a cryptographic hash for the full, canonical JSON content of each of the four records individually. The hashing algorithm MUST be SHA-256.
4.  The system SHALL create a `chain_hash`. This is generated by concatenating the individual SHA-256 hashes of the records in their exact chronological order (Canonical Request, Risk Evaluation, Authorization, Execution) and then computing a final SHA-256 hash of the concatenated string.
5.  The system SHALL perform all verification checks defined in the Decision Logic (Section 8).
6.  Upon successful verification, the system SHALL generate at least one cryptographic signature for the `attestation_record`. The signature payload is the `chain_hash`. The signing algorithm MUST be ECDSA with P-256 and SHA-256.
7.  The system SHALL construct the final `attestation_record.json` object, populating it with all required IDs, the record hashes, the `chain_hash`, verification check results, and the signature(s).
8.  The completed `attestation_record` SHALL be passed as the output to the next protocol step (Audit Ledger).

## 8. Decision Logic

This step is primarily procedural. However, it includes a critical verification gate. The process only proceeds if all verification checks pass. 

| Check Name | Logic | Action on Pass | Action on Fail |
|---|---|---|---|
| `ID_REFERENCE_INTEGRITY` | Verify that `request_id` is identical across all retrieved records. | Proceed | Fail (Error Code: A8-01) |
| `AUTHORIZATION_MATCH` | Verify that the `authorization_match` field in the `execution_record` is `true`. | Proceed | Fail (Error Code: A8-02) |
| `EXECUTION_STATUS_CHECK` | Verify that the `execution_status` in the `execution_record` is `COMPLETED_SUCCESSFULLY`. | Proceed | Fail (Error Code: A8-03) |

## 9. Failure Conditions

| Error Code | Trigger | Required Action |
|---|---|---|
| `A8-01` | One or more retrieved records do not share the same `request_id`. | The attestation process MUST halt. A critical error SHALL be logged indicating a potential chain tampering or data corruption event. Manual investigation is required. |
| `A8-02` | The `execution_record` indicates a mismatch between the authorized and executed action. | The attestation process MUST halt. This signifies a severe policy violation or system failure. An alert SHALL be raised. |
| `A8-03` | The `execution_record` indicates the action did not complete successfully. | The attestation process MUST halt. A failed execution cannot be attested as a valid, completed chain. The error state from the execution step should be propagated. |
| `A8-04` | Failure to generate a required cryptographic signature (e.g., key unavailable). | The process MUST NOT produce an `attestation_record`. An error SHALL be logged detailing the signature failure. |

## 10. Security Considerations

- **Cryptographic Agility:** While SHA-256 and ECDSA P-256 are the current standards, the system SHOULD be designed to allow for future upgrades to stronger algorithms.
- **Key Management:** The private key(s) used for signing the `attestation_record` MUST be stored in a hardware security module (HSM) or equivalent secure enclave. Access to this key must be strictly controlled and audited.
- **Replay Attacks:** The inclusion of unique, monotonically increasing IDs (`request_id`, `attestation_id`) and timestamps helps prevent the replay of old attestations.
- **Chain Integrity:** The `chain_hash` is the primary mechanism for ensuring the integrity of the decision sequence. Any modification to a prior record will result in a `chain_hash` mismatch, invalidating the attestation.

## 11. Audit Requirements

The full, final `attestation_record`, including all hashes and signatures, MUST be logged and stored in a way that is immutable and tamper-evident. This record is a cornerstone of the audit trail and provides the definitive proof of a transaction's lifecycle.

## 12. Dependencies

- **Upstream:** This step is directly dependent on the successful completion of **Step 07: Execution**. It cannot begin until a final `execution_record` is available.
- **Downstream:** **Step 09: Audit Ledger** is dependent on this step. The `attestation_record` is the primary input for the audit ledger, which is responsible for its long-term, immutable storage.

## 13. Example Flow

Continuing the example of the $48,250 wire transfer to Meridian Industrial Supply:

1.  **Initiation:** The Execution step completes, producing an `execution_record` with `execution_id: "exec-a7b3c1f0-9d2e-4f8a-8c7b-1d0e6a2f9b3d"` and `execution_status: "COMPLETED_SUCCESSFULLY"`.
2.  **Record Retrieval:** The system retrieves the four records associated with `request_id: "req-abc123xyz"`.
3.  **Hashing:**
    - `hash(canonical_request)` -> `0a1b2c...`
    - `hash(risk_evaluation)` -> `3d4e5f...`
    - `hash(authorization_record)` -> `6g7h8i...`
    - `hash(execution_record)` -> `9j0k1l...`
4.  **Chain Hash:**
    - `concatenated_hashes` = `"0a1b2c...3d4e5f...6g7h8i...9j0k1l..."`
    - `chain_hash` = `SHA256(concatenated_hashes)` -> `e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855`
5.  **Verification:** All checks pass: `ID_REFERENCE_INTEGRITY` is true, `AUTHORIZATION_MATCH` is true, and `EXECUTION_STATUS_CHECK` is true.
6.  **Signature:** The RIO system's HSM signs the `chain_hash` (`e3b0...b855`) with its private key.
7.  **Output Generation:** The final `attestation_record` is created:

```json
{
  "attestation_id": "attest-f4b3c1f0-9d2e-4f8a-8c7b-1d0e6a2f9b3d",
  "request_id": "req-abc123xyz",
  "risk_evaluation_id": "risk-def456uvw",
  "authorization_id": "auth-ghi789rst",
  "execution_id": "exec-a7b3c1f0-9d2e-4f8a-8c7b-1d0e6a2f9b3d",
  "record_hashes": {
    "canonical_request_hash": "0a1b2c...",
    "risk_evaluation_hash": "3d4e5f...",
    "authorization_record_hash": "6g7h8i...",
    "execution_record_hash": "9j0k1l..."
  },
  "chain_hash": "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855",
  "verification_checks": {
    "ID_REFERENCE_INTEGRITY": "PASSED",
    "AUTHORIZATION_MATCH": "PASSED",
    "EXECUTION_STATUS_CHECK": "PASSED"
  },
  "signatures": [
    {
      "signer_id": "RIO_SYSTEM_HSM_01",
      "signature": "3045022100...",
      "algorithm": "ECDSA-SHA256",
      "timestamp": "2026-03-24T18:05:00Z"
    }
  ]
}
```
This record is then passed to the Audit Ledger.
