# RIO Protocol — Governance Learning Specification

**Version:** 1.0.0
**Status:** Normative
**Protocol Reference:** 10 — Learning Protocol

---

## 1. Overview

The learning protocol is the feedback loop that makes the RIO system adaptive. It analyzes completed decision chains to identify patterns, anomalies, and improvement opportunities across risk evaluation accuracy, policy effectiveness, and authorization efficiency.

The learning system operates under a fundamental constraint: **all policy and risk model updates generated by the learning system MUST be authorized through the RIO Protocol itself.** The learning system proposes changes; humans approve them. The system MUST NOT modify its own governance rules without explicit human authorization.

---

## 2. Learning Inputs

The learning system consumes the following data from the RIO Protocol ledger and receipt store:

| Input | Source | Purpose |
|-------|--------|---------|
| Completed receipts | Receipt store | Analyze full decision chains for outcome patterns |
| Denied requests | Authorization records | Identify denial patterns and potential false positives |
| Expired authorizations | Authorization records | Detect workflow bottlenecks and timeout issues |
| Execution failures | Execution records | Correlate risk predictions with actual failure rates |
| Authorization response times | Timeline data in receipts | Measure human response latency and identify delays |
| Risk score accuracy | Risk evaluation vs. execution outcome | Compare predicted risk with actual outcome (predicted high risk but executed without issue, or predicted low risk but failed) |
| Policy trigger frequency | Policy flags in risk evaluations | Identify policies that trigger too often (overly broad) or never (dead rules) |
| Anomaly indicators | Cross-record analysis | Detect unusual patterns in request volume, timing, or parameters |

### 2.1 Data Collection

The learning system MUST collect data passively from the ledger. It MUST NOT inject requests, modify records, or interfere with active decision chains. Data collection operates on completed chains only — never on in-progress decisions.

## 3. Learning Event Schema

When the learning system identifies a pattern, anomaly, or improvement opportunity, it creates a `learning_event` record:

```json
{
  "event_id": "string (UUID v4)",
  "event_type": "enum",
  "source_records": ["array of receipt_ids"],
  "detected_at": "string (ISO 8601 UTC)",
  "confidence_score": "number (0.0 - 1.0)",
  "category": "string",
  "description": "string",
  "evidence": {
    "sample_size": "integer",
    "time_range": {
      "start": "string (ISO 8601)",
      "end": "string (ISO 8601)"
    },
    "statistical_summary": "object"
  },
  "recommended_action": {
    "action_type": "enum",
    "target": "string",
    "proposed_change": "object",
    "expected_impact": "string",
    "rollback_plan": "string"
  },
  "status": "enum",
  "reviewed_by": "string or null",
  "reviewed_at": "string (ISO 8601) or null",
  "implementation_receipt_id": "string (UUID v4) or null"
}
```

### 3.1 Event Types

| Event Type | Description |
|-----------|-------------|
| `pattern_detected` | Recurring pattern identified across multiple decision chains |
| `anomaly_detected` | Unusual deviation from established baselines |
| `policy_recommendation` | Suggestion to create, modify, or retire a policy rule |
| `risk_model_update` | Proposed adjustment to risk scoring weights or thresholds |
| `threshold_adjustment` | Recommended change to authorization thresholds |

### 3.2 Status Values

| Status | Description |
|--------|-------------|
| `pending_review` | Awaiting human review |
| `approved` | Human has approved the recommended action |
| `rejected` | Human has rejected the recommendation |
| `implemented` | Approved change has been applied through the RIO Protocol |
| `superseded` | A newer learning event has replaced this one |

## 4. Learning Categories

### 4.1 Risk Model Tuning

The learning system analyzes the correlation between predicted risk scores and actual outcomes to improve risk evaluation accuracy.

**Inputs:** Risk evaluation records paired with execution outcomes.

**Analysis:**
- False positive rate: requests scored as high risk that executed without issue.
- False negative rate: requests scored as low risk that resulted in failures or incidents.
- Risk factor weight accuracy: which factors are most predictive of actual outcomes.

**Output:** Proposed adjustments to risk factor weights, category thresholds, or scoring formulas.

### 4.2 Policy Effectiveness

The learning system evaluates how well existing policies serve their intended purpose.

**Metrics:**
- **Trigger frequency:** Policies that trigger on more than 80% of requests may be overly broad.
- **Never-triggered policies:** Policies that have not triggered in 90 days may be dead rules.
- **Override rate:** Policies whose constraints are frequently overridden by authorizers may need revision.
- **False denial rate:** Policies that deny requests that are subsequently re-submitted and approved indicate overly restrictive rules.

### 4.3 Authorization Efficiency

The learning system measures the human authorization workflow to identify bottlenecks.

**Metrics:**
- Mean time to authorization decision.
- Authorization timeout rate (expired before decision).
- Authorizer workload distribution.
- Peak authorization request times.

**Output:** Recommendations for threshold adjustments, delegation rules, or workflow changes.

### 4.4 Anomaly Detection

The learning system monitors for deviations from established baselines.

**Detection patterns:**
- Sudden increase in request volume from a single agent.
- Requests at unusual times (outside business hours for time-sensitive operations).
- Repeated requests for the same action after denial.
- Parameter values significantly outside historical ranges.
- New action types not previously seen from a given requester.

### 4.5 Operational Metrics

The learning system tracks system-level performance indicators.

| Metric | Description | Target |
|--------|-------------|--------|
| Mean decision chain duration | Time from request to receipt | < 300 seconds |
| Risk evaluation latency | Time for risk scoring | < 2 seconds |
| Policy evaluation latency | Time for policy matching | < 1 second |
| Ledger write latency | Time to record a ledger entry | < 500 milliseconds |
| System availability | Uptime of the RIO control plane | 99.9% |
| Throughput | Decision chains processed per minute | Varies by deployment |

## 5. Policy and Risk Update Principles

This section defines the rules governing how learning-generated recommendations become actual system changes.

### 5.1 Fundamental Rule

**ALL policy updates generated by the learning system MUST go through the RIO Protocol authorization process.** This is the meta-governance principle. The system that governs AI actions MUST itself be governed.

### 5.2 Update Workflow

```
Learning System detects pattern
    → Creates learning_event (status: pending_review)
    → Generates canonical_request for the proposed change
        action_type: learn.update.policy OR learn.update.risk_model
    → Risk evaluation of the proposed change
    → Human authorization required (ALWAYS — no exceptions)
    → If approved: implementation executed
    → Attestation of the change
    → Receipt generated
    → Ledger entry recorded
    → learning_event status updated to "implemented"
    → implementation_receipt_id linked
```

### 5.3 Update Constraints

1. **No automated policy changes.** The learning system MUST NOT modify active policies without human approval. No exceptions.
2. **Rollback required.** Every proposed change MUST include a rollback plan that can restore the previous state.
3. **Gradual rollout.** Risk model changes SHOULD be deployed incrementally (e.g., shadow mode before enforcement).
4. **Audit trail.** Every change MUST produce a complete decision chain in the ledger, linking the learning event to the implementation receipt.
5. **Confidence threshold.** Recommendations with confidence_score below 0.7 SHOULD be flagged for additional review.

---

## 6. Learning Boundaries

The learning system MUST NOT:

1. **Modify active policies without authorization.** No policy change, regardless of confidence, may be applied without human approval through the RIO Protocol.
2. **Bypass the authorization chain.** The learning system is subject to the same governance rules as any other system component.
3. **Reduce security controls.** Recommendations that would lower risk thresholds, remove authorization requirements, or weaken policy constraints MUST be flagged as high-risk changes requiring elevated authorization.
4. **Override human decisions.** If a human authorizer consistently overrides a policy, the learning system may recommend a policy change, but it MUST NOT automatically adjust the policy.
5. **Access raw request parameters.** The learning system operates on aggregated and anonymized data. It MUST NOT have access to sensitive business data within request parameters unless explicitly authorized.
6. **Self-modify.** The learning system MUST NOT modify its own analysis algorithms, confidence thresholds, or detection rules without human authorization.

## 7. Feedback Loop Architecture

The data flow for the governance learning system follows this path:

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Ledger ──► Learning Engine ──► Recommendations         │
│                                       │                 │
│                                       ▼                 │
│                              Canonical Request          │
│                                       │                 │
│                                       ▼                 │
│                              Risk Evaluation            │
│                                       │                 │
│                                       ▼                 │
│                              Human Authorization        │
│                                       │                 │
│                                       ▼                 │
│                              Implementation             │
│                                       │                 │
│                                       ▼                 │
│                    Policy/Risk Model Updates ──► Ledger  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 7.1 Data Flow Rules

1. The learning engine reads from the ledger. It MUST NOT write to the ledger directly.
2. Recommendations flow through the standard RIO Protocol pipeline.
3. Implemented changes are recorded in the ledger, creating a closed feedback loop.
4. The learning engine can observe the effects of its own recommendations in subsequent decision chains.

## 8. Metrics and Reporting

The learning system MUST produce the following standard reports:

### 8.1 Standard Metrics

| Metric | Description | Reporting Frequency |
|--------|-------------|-------------------|
| False positive rate | % of high-risk scores that resulted in successful execution | Weekly |
| False negative rate | % of low-risk scores that resulted in failures | Weekly |
| Mean authorization time | Average time from request to authorization decision | Daily |
| Policy coverage | % of action types covered by at least one policy | Monthly |
| Risk prediction accuracy | Correlation between predicted risk and actual outcome | Monthly |
| Anomaly detection rate | Number of anomalies detected per 1,000 decision chains | Weekly |
| Recommendation acceptance rate | % of learning recommendations approved by humans | Monthly |

### 8.2 Report Format

Reports MUST be generated as structured JSON documents and stored in the ledger as `monitor.report` action types. Reports MAY additionally be exported as PDF or CSV for human review.

## 9. Implementation Considerations

### 9.1 Batch vs. Streaming Analysis

- **Batch analysis** is suitable for policy effectiveness reviews, risk model tuning, and operational metrics. Run daily or weekly.
- **Streaming analysis** is required for anomaly detection and real-time operational monitoring. Process events as they are recorded in the ledger.

Implementations SHOULD support both modes. Batch analysis provides deeper statistical insights; streaming analysis provides timely anomaly detection.

### 9.2 Data Retention for Learning

- The learning system MUST have access to at least 90 days of historical decision chains for pattern detection.
- Risk model tuning SHOULD use at least 180 days of data for statistical significance.
- Anomaly detection baselines SHOULD be recalculated monthly.

### 9.3 Privacy Considerations

- The learning system MUST operate on aggregated data wherever possible.
- Individual request parameters SHOULD be anonymized before analysis.
- Learning events MUST NOT contain personally identifiable information (PII) unless the PII is essential to the recommendation and the learning event is subject to the same access controls as the source records.
