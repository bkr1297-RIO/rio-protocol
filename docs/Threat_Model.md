# Threat Model — RIO Execution Governance Infrastructure

**Version:** 2.0
**Status:** Reference implementation (143 tests, 0 failures)

## Purpose

This document identifies the execution-layer threats that RIO is designed to mitigate in autonomous AI environments. RIO operates at the boundary between intent and action — the point where an AI system's recommendation becomes a real-world consequence. The threat model is scoped to this boundary.

## Design Principle

> **Fail-closed.** If any condition cannot be positively verified, the execution gate remains locked. The system defaults to denial, not permission.

## Threat Categories

### T1. Unauthorized Execution

**Attack:** An agent or system attempts to execute a real-world action without proper authorization.

**Mitigation:** The execution gate is fail-closed. No action can proceed without a valid, signed Execution Token. The token must pass four checks: signature validity, timestamp within TTL, unused nonce, and kill switch status. If any check fails, the gate remains locked and a denial receipt is generated.

**Evidence:** Core Pipeline test suite — 57 tests verify that execution without a valid token is rejected.

### T2. Replay Attacks

**Attack:** An attacker captures a valid Execution Token and resubmits it to execute the same action again.

**Mitigation:** Every Execution Token contains a single-use nonce. The nonce registry tracks all used nonces. A token with a previously used nonce is rejected at the execution gate.

**Evidence:** Nonce registry tests verify that resubmitted tokens are rejected. The live gateway returns `nonce_reuse_blocked` counts in its stats endpoint.

### T3. Stale Authorization

**Attack:** An approval granted hours or days ago is used to execute an action that is no longer appropriate.

**Mitigation:** Execution Tokens carry a TTL (default 300 seconds). The execution gate checks the token timestamp against the current time. Expired tokens are rejected.

**Evidence:** TTL expiry tests confirm that tokens beyond the TTL window are rejected.

### T4. Audit Tampering

**Attack:** An operator modifies ledger entries after the fact to conceal unauthorized actions or alter the historical record.

**Mitigation:** The ledger is a signed hash chain. Each entry `En` contains:

```
Hn = SHA256(En.data + H(n-1))
```

Any modification to any entry invalidates all subsequent hashes. Each entry also carries its own Ed25519/ECDSA signature. The independent verifier can detect tampering without access to the runtime.

**Evidence:** Ledger verification tests confirm that tampered entries are detected. The simulator's `--generate-tampered-ledger` mode demonstrates detection of a single modified row in a 3-entry chain.

### T5. Receipt Forgery

**Attack:** An attacker fabricates a receipt to create a false audit record showing that an action was properly authorized when it was not.

**Mitigation:** Every receipt is signed with Ed25519 or ECDSA. The three-hash binding (`intent_hash`, `action_hash`, `verification_hash`) cryptographically ties what was intended, what was executed, and what was observed. Forgery is detectable by any party with access to the public key.

**Evidence:** Receipt verification tests (7 checks) confirm that modified receipts fail signature validation.

### T6. Silent Denials

**Attack:** The system blocks an action but does not record the denial, creating a gap in the audit trail.

**Mitigation:** Denials are first-class in RIO. Blocked or denied actions generate denial receipts with the same structure as approval receipts. Denial receipts are recorded in the ledger. The audit trail covers every decision, not just successes.

**Evidence:** Denial receipt tests verify that blocked actions produce signed, ledger-recorded denial receipts.

### T7. Policy Bypass

**Attack:** An agent or operator circumvents the policy engine to execute an action that should have been blocked or required human approval.

**Mitigation:** The policy engine is evaluated before the execution gate. The execution gate only accepts tokens generated by the authorization stage, which requires policy evaluation. There is no path from intent to execution that bypasses policy.

**Evidence:** The 8-stage pipeline is enforced sequentially. Tests verify that actions requiring approval cannot proceed without it.

### T8. Key Compromise

**Attack:** An attacker obtains the signing key and can forge tokens and receipts.

**Mitigation (partial):** Keys are currently software-managed. Key rotation is supported. Compromised keys can be revoked. However, HSM integration for hardware-protected keys is future work.

**Limitation:** This is an acknowledged gap. Software-managed keys are vulnerable to host compromise. HSM integration would close this gap.

### T9. Denial of Service

**Attack:** An attacker floods the system with requests to prevent legitimate actions from being processed.

**Mitigation (partial):** Rate limiting is implemented at the gateway level. However, sophisticated DDoS attacks are outside the protocol scope and require infrastructure-level mitigation.

### T10. Insider Threat

**Attack:** A trusted operator with access to the runtime modifies the system to bypass governance controls.

**Mitigation:** The independent verifier operates outside the runtime. Any party with access to the public key and the ledger can verify the integrity of the audit trail. Insider modifications to the runtime cannot produce receipts that pass independent verification unless the signing key is also compromised.

## Scope Boundaries

RIO's threat model is explicitly scoped to the **execution layer**. The following are out of scope by design:

| Domain | Why Out of Scope |
|--------|-----------------|
| Data governance (training data quality, bias) | RIO governs actions, not training |
| Model transparency / explainability | RIO records what happened, not why the model recommended it |
| Accuracy / robustness | RIO does not evaluate model quality |
| Content safety / guardrails | RIO governs execution, not generation |
| Full lifecycle logging beyond execution | RIO covers the execution boundary |
| GDPR retention / minimization | Requires domain-specific data governance |
| HSM key management | Future work (acknowledged limitation) |

> **RIO governs what AI systems do, not what they are trained on or what they say.**
